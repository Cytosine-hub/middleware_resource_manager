# Wiki 编译质量问题清单

**发现日期**: 2026-06-12
**编译文档**: 100_TongWeb_V7.0集群管理指南_7049_M10A01.pdf（5.9MB）
**编译结果**: PARTIAL（覆盖率 100%，48 个页面，13 个过度压缩页面）

---

## 1. 标题路径冗余

**现象**：wiki 页面的内容小标题带上了完整的路径前缀，与主标题重复。

**示例**：
- 主标题已经是「集群管理/服务器实例管理」
- 但内容里的小标题还写「集群管理/服务器管理/录入服务器实例管理」
- 实际应该只写「录入服务器实例管理」，因为父级路径已经在主标题中体现了

**影响**：标题冗长，阅读体验差，信息重复。

**修复方向**：在生成页面内容时，小标题只保留当前层级的标题，不重复父级路径。

---

## 2. 标题软件名冗余

**现象**：每个 wiki 页面的标题都带上了软件名，如「TongWeb 7 集群管理 - XXX」。

**示例**：
- 当前标题：「TongWeb V7.0 集群管理 - License与节点管理」
- 期望标题：「License与节点管理」
- 软件信息（TongWeb V7.0）已经通过 category 和 software 标签标识了，标题不需要重复

**影响**：
- 标题冗长
- 前端列表显示不友好
- 搜索时标题噪音大

**修复方向**：在 LLM 生成页面时，prompt 中明确要求标题不带软件名前缀，软件信息由标签承载。

---

## 3. 缺少增量编译逻辑

**现象**：当前编译是"全量替换"，不是"增量补充"。

**当前流程**：
```
outline → section_facts → page_plan → page_generation → quality_gate
```

**问题**：
- PARTIAL 状态的编译已保存了 35 个页面，但 `ingested=false`
- 如果重新编译，会删除旧页面、重新走完整流程
- LLM 生成有随机性，重新编译结果可能不同
- 无法只补充缺失的章节

**用户期望**：
- 编译是"全量替换"，不是"增量补充"
- 如果要实现增量补充，需要：
  1. 读取上次质量报告的 `missingSections`（缺 48 个章节）
  2. 只针对缺失章节生成补充页面
  3. 与现有页面合并（更新已有页面、新增缺失页面）
  4. 重新跑质量门禁

**技术细节**：
- 过程产物已持久化：`section_facts`（250KB）、`page_plan`（12KB）、`quality_report`
- 增量编译可以从 `section_facts` 开始，跳过最耗时的步骤
- 只针对缺失章节重新生成页面，而不是全量重跑

**修复方向**：在 `ingestPlanned` 中增加"增量模式"分支，读取上次的 `section_facts` 和 `page_plan`，只重生成缺失章节。

---

## 4. 过度压缩页面无法单独重编译

**现象**：覆盖率 100% 但存在过度压缩页面（content 太短，塞了太多章节）。

**示例**：
- Task 18：覆盖率 100%（397/397），但有 13 个过度压缩页面
- 质量门禁状态为 PARTIAL，因为存在过度压缩页面
- 当前无法只重编译这 13 个页面，只能全量重跑

**技术细节**：
- 质量门禁判断逻辑：`requiredSectionsInPage >= 2 && content.length < requiredSectionsInPage * 300`
- 即：如果一个页面覆盖了 2 个以上必需章节，但内容长度不足 `章节数 × 300 字符`，则判定为过度压缩
- `quality_report.overCompressedPages` 记录了过度压缩的页面列表

**修复方向**：
- 读取 `quality_report.overCompressedPages` 识别过度压缩页面
- 只重生成这些页面（保留其他正常页面）
- 合并到现有页面，重新跑质量门禁

## 5. Lint 结果过期显示

**现象**：点击 Lint 标签时显示旧的 lint 结果，即使 wiki 内容已变。

**原因**：
- `loadLintResults()` 从数据库加载持久化的 lint 结果（`/api/wiki/lint/results`）
- 不是实时计算的
- 旧的 lint 结果没有自动清理

**流程**：
```
点击 Lint 标签 → loadLintResults() → 从数据库读取旧结果 → 显示过期数据
点击运行 Lint → runLint() → 重新检测并覆盖旧结果 → 旧问题消失
```

**影响**：用户看到的 lint 结果可能不准确，需要手动点击"运行 Lint"才能看到真实状态。

**修复方向**：点击 Lint 标签时自动运行 lint（调用 `runLint()`），而不是加载旧结果。

## 6. 图谱边数爆炸，视觉显示 1000+ 节点

**现象**：数据库只有 48 个 wiki 页面，但图谱显示 1000+ 个节点。

**原因分析**：
- 48 个页面全属同一软件（TongWeb）、同一分类（中间件）、同一来源文档
- 图谱使用 5 信号评分，为同一组的页面两两创建边：
  - 信号1（直接链接）：24 条边
  - 信号2（同软件）：C(48,2) = 48×47/2 = **1128 条边**
  - 信号4（同来源）：C(48,2) = 48×47/2 = **1128 条边**
  - 信号3（同分类）：仅 OVERVIEW 页面之间，约 0-1 条边
- 总计约 **2280 条边**

**视觉问题**：
- `force-graph` 库渲染所有 2280 条边
- 边的交叉点在视觉上看起来像额外的"节点"
- 用户误以为有 1000+ 个节点

**修复方向**：
- 只显示权重大于阈值的边（如 weight >= 3）
- 或者限制每个节点的最大边数
- 或者在后端生成图谱时就过滤低权重边

## 7. 向量化策略调整：源文档向量化，wiki 页面不向量化

**当前做法**：
- 源文档（PDF 原文 253KB）存储在 `wiki_sources.content`，**不向量化**
- Wiki 页面（48 个）存储在 `wiki_pages`，**向量化**（使用 title + summary）

**用户观点**：
- Wiki 页面是 LLM 生成的摘要，丢失了原文细节
- 向量化 wiki 页面是重复工作
- 应该向量化源文档原文，搜索时直接命中原文片段，更精准

**期望做法**：
- 源文档向量化（按 chunk 切分后向量化）
- Wiki 页面不向量化
- 搜索时直接从源文档原文中检索，保留完整上下文

**影响**：
- 需要修改 `IngestAgent.vectorizePages()` 逻辑
- 需要新增源文档向量化逻辑（参考 knowledge 模块的 chunk 切分和向量化）
- 前端搜索结果展示需要调整（从展示 wiki 页面改为展示源文档片段）
